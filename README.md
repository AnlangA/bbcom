# bbcom

项目概览
---------
`bbcom` 是一个基于 Tauri + 前端（React / Vue / 等）与 Rust 后端的混合桌面应用模版，主要用于串口/设备通信与可视化控制场景。项目聚焦于稳定的串口数据流处理、前后端通信和可扩展的模块化架构。该仓库包含前端资产、Tauri/Rust 后端以及若干开发与测试约定。

本 README 的目的
---------------
- 为你（开发者）提供快速上手与常用命令。
- 描述项目结构与主要模块职责。
- 概述本仓库的开发/构建/测试流程与协作规范。

关于 `AGENTS.md`
-----------------
仓库原先包含一份名为 `AGENTS.md` 的项目 AI 开发指南（包含架构约定、模块模板、测试策略、提交规范等）。根据当前项目维护决策，该文件已从仓库中移除。文件中的关键约定与行为准则已在本 README 与各模块内的局部文档中整合，请参阅对应模块的说明（若模块下存在同名 `AGENTS.md`，优先查阅模块内文档）。

先决条件
--------
- Rust（建议 stable）
- Node.js（v16+ 推荐）与包管理器（`pnpm` / `npm` / `yarn`）
- Tauri CLI（用于构建与打包，参见 Tauri 官方文档）
- 串口设备访问权限（开发机器上需要相应驱动与权限）

常用目录（高层）
----------------
- `src-tauri/` — Tauri + Rust 后端源码与 `Cargo.toml`
- `src/` 或 `frontend/` — 前端代码（取决于具体前端框架）
- `packages/` — (若存在) 多包/monorepo 子包
- `assets/` — 静态资源（图标、示例数据等）
- `tests/` — 集成 / E2E 测试脚本
- `docs/` — 项目级文档与设计文档（如果存在）

典型文件
- `README.md` — 本文件
- `LICENSE` — 许可证
- `package.json` / `pnpm-workspace.yaml` — 前端依赖与脚本
- `src-tauri/Cargo.toml` — 后端依赖与设置

快速开始（开发）
----------------
1. 克隆仓库并进入目录：
   - 使用你熟悉的 git 命令：`git clone <repo>` 然后 `cd bbcom`
2. 安装前端依赖：
   - 例如使用 `pnpm`：`pnpm install`
3. 启动开发模式（前端热重载 + 后端热重载）：
   - 前端：运行你项目的前端 dev 脚本，例如 `pnpm dev`（或 `npm run dev` / `yarn dev`）
   - Tauri/Rust：在另一个终端运行 `cargo build` 或使用 Tauri 的 dev 命令（例如 `pnpm tauri dev`，依项目脚本而定）
4. 打开开发工具并连接串口设备（若需要）

构建与打包
-----------
- 前端构建：运行前端构建脚本（例如 `pnpm build`）
- Tauri 打包：在前端构建之后运行 Tauri 打包流程（例如 `pnpm tauri build` 或 `cargo tauri build`）
- 生成的可执行文件会在 Tauri 指定的输出目录中（参见 `src-tauri/tauri.conf.json`）

测试
----
- 单元测试（Rust）：在 `src-tauri` 目录运行 `cargo test`
- 前端测试：使用框架推荐的测试命令（例如 `pnpm test`）
- 集成 / E2E：若有 `tests/` 或使用 Cypress / Playwright，请运行相应脚本

串口与设备通信（注意事项）
------------------------
- 串口通常由后端（Rust）打开并做二进制/协议解析，前端通过 Tauri command 或事件订阅接收数据。
- 推荐在后端实现：可配置的波特率、数据位、停止位、奇偶校验及重连逻辑。
- 渲染性能：前端不应对每个字节做 DOM 更新，建议使用数据合并 / 请求动画帧（requestAnimationFrame）等批量渲染策略。
- 权限：打包平台（macOS / Linux / Windows）对串口设备访问可能有系统权限要求，打包与部署时务必测试目标平台。

开发约定（高层）
----------------
- 提交信息：遵循 Conventional Commits（例如 `feat:`, `fix:`, `chore:`）
- 分支策略：主干（`main`）/ 开发（`dev`）/ 功能分支（`feat/...`）
- 每个模块若需要规范性说明，放置局部 `AGENTS.md`（模块级开发指南）
- 代码审查：PR 必须包含变更描述、测试说明与影响范围

安全与发布注意
---------------
- 不要将敏感信息（API Key、私钥）硬编码到仓库中。使用环境变量或运行时注入。
- 打包时核查依赖许可与体积，审计本地 native 依赖（例如串口库）。

调试建议
--------
- 后端：使用 `cargo test`、`cargo run` 并在必要处添加日志（建议使用 `env_logger` / `tracing`）
- 前端：使用浏览器开发者工具或 Tauri 提供的调试窗口
- 串口问题：在开发机上使用串口调试工具（例如 `minicom` / `screen` / `PuTTY`）确认设备行为

贡献指南
--------
1. Fork 仓库并基于 `dev` 分支创建你的特性分支：`feat/xxx`。
2. 保持提交小而清晰，使用 `Conventional Commits`。
3. 提交 PR 并在 PR 描述中说明变更点与测试方式。
4. 代码审查通过并 CI 全绿后合并到 `dev` 或 `main`。

常见问题（FAQ）
--------------
- Q: 开发时显示找不到串口 / 权限被拒绝？
  - A: 检查操作系统权限（macOS 的系统设置、Linux 的 udev 规则或 Windows 驱动），并确保当前用户对设备文件有读写权限。
- Q: 为什么前端卡顿当数据量高时？
  - A: 避免对每个接收到的字节做一次 DOM 更新，采用批量合并与节流策略。
